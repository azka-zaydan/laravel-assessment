## Notes

### Why Strategy appears in three places

Strategy is the single most load-bearing pattern in this codebase because the domain has three independent axes of variation: *where restaurant data comes from*, *what kind of Telegram update arrived*, and *which bot command the user invoked*. Each axis needs to be extensible (add a new provider, handler, or command) without touching the dispatcher. Applying Strategy to all three sites produces three small, closed registries (`RestaurantServiceProvider`, `MessageDispatcher`, `CommandRegistry`) each of which adds a new algorithm by implementing one interface â€” no `if/else` chains, no switch statements.

### Why the 2FA challenge token is a separate JWT format

When a user with 2FA enabled calls `POST /api/login`, issuing a full Passport access token would grant unrestricted API access before the second factor is verified â€” defeating the purpose of 2FA. Instead, `ChallengeTokenService` issues a compact, short-lived JWT (5-minute TTL) signed with a separate `JWT_CHALLENGE_SECRET`. This token carries only a `sub` (user ID) and a `type: challenge` claim, which `TwoFactorController@verify` validates before finally issuing the real Passport token. The two token formats are intentionally incompatible: `auth:api` middleware rejects challenge tokens because they are not Passport-issued; the verify endpoint rejects Passport tokens because they lack the `challenge` type claim. The Adapter pattern in `TelegramBotService` and the Observer pattern in `ApiLogObserver` follow a similar principle of keeping formats and concerns strictly separated.

### Repository + Write-through cache

`RestaurantRepository` combines the Repository and write-through cache patterns rather than keeping them in separate layers. This is a deliberate trade-off: because the Zomato API is effectively dead and `MockProvider` is the source of truth, the "write" side of write-through is an upsert into Postgres that acts as a persistent mock store. The Redis layer is purely a hot-cache on top. Separating this into a dedicated `CacheRepository` decorator would add indirection without benefit given that there is only one concrete provider in active use. The boundary is still clean â€” `RestaurantService` calls only `RestaurantRepository` and has no awareness of Redis.

### Pipeline and middleware ordering

The `LogApiRequest` middleware is appended to the `api` group via `appendToGroup` (rather than declared as a route middleware) for two reasons: (1) it must run *after* `auth:api` resolves the user so that `user_id` can be logged, and (2) its `terminate()` method dispatches `LogApiRequestJob` *after* the HTTP response has been sent to the client, so logging never adds measurable latency. This is a precise fit for Laravel's Pipeline/Middleware pattern where `terminate()` provides a post-response hook distinct from the main request pipeline.
